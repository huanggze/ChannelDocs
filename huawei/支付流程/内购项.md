## 一、消耗型商品购买流程

消耗型商品典型购买流程如下：

1. 用户发起购买后，应用客户端向HMS Core（APK）发起购买请求，携带商品ID、商品类型等信息。
2. HMS Core（APK）生成订单，返回IAP收银台。
3. 应用客户端启动IAP收银台。
   > 说明：若使用createPurchaseIntentWithPrice接口发起支付，您需要使用IAP公钥对接口返回的paymentData数据进行签名校验。验签成功后再启动IAP收银台。
4. 用户完成支付后，HMS Core（APK）返回购买数据InAppPurchaseData（包含是否购买成功、订单ID、商品ID、购买Token等）及其签名数据。
5. 应用客户端向应用服务器上报购买数据及其签名数据。
6. （建议）若您的应用对安全性要求较高，可通过服务端Order服务验证购买Token接口，向IAP服务器查询购买数据及其签名数据，通过此接口可进一步确认订单的准确性。验证购买Token时，需携带购买Token和商品ID信息。
7. （建议）IAP服务端返回购买数据及其签名数据。
8. 应用服务器使用IAP公钥验证购买数据签名。为避免资金损失，您在验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。
9. 验证成功后，发放商品，记录商品的发货状态（购买Token和商品ID）。
   > 注意：务必保存已发货商品的购买Token，后续即使消耗失败也可从您的服务器获取商品的发货状态，避免重复发货的情况。
10. 应用服务器返回发货结果给应用客户端。
11. 应用通知IAP服务器更新商品发货状态。请务必确保发货成功后再调用本步骤。发货成功后应用客户端向HMS Core（APK）发送consumeOwnedPurchase请求以消耗该商品，以此通知IAP服务器更新商品的发货状态。发送consumeOwnedPurchase请求时，需携带购买数据中的purchaseToken。您也可使用服务端Order服务确认购买接口消耗商品，用于替换IAP客户端consumeOwnedPurchase接口。
    > 注意：如果您在消耗之后再进行发货，将存在以下异常场景：您的应用发起消耗请求之后，华为应用内服务器已消耗成功但因网络异常等原因没有成功将结果回调给您的应用，该情况将造成服务器已经消耗但是应用没有发货的场景。此场景后续很难处理，为了避免不必要的问题，您需要在发货成功之后再执行消耗操作。
12. HMS Core（APK）向应用返回消耗结果。应用成功执行消耗之后，IAP服务器会将相应商品重新设置为可购买状态，用户即可再次购买该商品。

> 说明：当前流程对单机游戏同样适用，只需要将流程图中服务端的处理放在客户端即可。消耗型商品请务必实现消耗型商品的补单流程，避免掉单问题发生。

## 二、非消耗型商品购买流程

非消耗型商品典型购买流程如下：

1. 用户发起购买后，应用客户端向HMS Core（APK）发起购买请求，携带商品ID、商品类型等信息。
2. HMS Core（APK）生成订单，返回IAP收银台。
3. 应用客户端启动IAP收银台。
   > 说明：若使用createPurchaseIntentWithPrice接口发起支付，您需要使用IAP公钥对接口返回的paymentData数据进行签名校验。验签成功后再启动IAP收银台。
4. 用户完成支付后，HMS Core（APK）返回购买数据InAppPurchaseData（包含是否购买成功、订单ID、商品ID、购买Token等）及其签名数据。
5. 应用客户端向应用服务器上报购买数据及其签名数据。
6. （建议）若您的应用对安全性要求较高，可通过服务端Order服务验证购买Token接口，向IAP服务器查询购买数据及其签名数据，通过此接口可进一步确认订单的准确性。验证购买Token时，需携带购买Token和商品ID信息。
7. （建议）IAP服务端返回购买数据及其签名数据。
8. 应用服务器使用IAP公钥验证购买数据签名。为避免资金损失，您在验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。
9. 验证成功之后需通过购买数据中的purchaseState字段判断商品的购买状态。当购买状态为已购买（purchaseState=0）时，发放相应商品。

> 说明：对于非消耗型商品，用户只需购买一次就可以永久拥有该商品。

## 三、订阅型商品购买流程

**首次购买流程：**

1. 用户发起购买后，应用客户端向HMS Core（APK）发起createPurchaseIntent购买请求，携带商品ID、商品类型等信息。
2. HMS Core（APK）生成订单，返回IAP收银台。
3. 应用客户端启动IAP收银台。
4. 用户完成支付后，HMS Core（APK）返回购买数据InAppPurchaseData（包含是否购买成功、订阅ID、商品ID、购买Token等）及其签名数据。
5. 应用客户端向应用服务器上报购买数据及其签名数据。
6. （建议）若您的应用对安全性要求较高，可通过服务端Subscription服务验证购买Token接口，向IAP服务器查询购买数据及签名数据，通过此接口可进一步确认订单的准确性。验证购买Token时，需携带购买Token和订阅ID信息。
7. （建议）IAP服务端返回购买数据及其签名数据。
8. 应用服务器使用IAP公钥验证购买数据签名。为避免资金损失，您在验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。
9. 验证成功后，向用户提供相应的商品服务。

**订阅仍生效期间，持续提供商品服务：**

1. 应用客户端向HMS Core（APK）发送obtainOwnedPurchases请求查询已购订阅型商品。
2. HMS Core（APK）返回用户所有已订阅且正生效的商品的购买数据InAppPurchaseData（包含是否购买成功、订阅ID、商品ID、购买Token等）以及每个购买数据的签名数据。
3. 应用客户端向应用服务器上报购买数据及其签名数据。
4. 应用服务器使用IAP公钥验证购买数据签名。为避免资金损失，您在验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。
5. 验证成功后，向用户提供相应的商品服务。

## 四、服务端 API 列表

来源：https://developer.huawei.com/consumer/cn/doc/HMSCore-Guides/server-dev-0000001050035027

| API 名称 | 说明 |
|---------|------|
| Order服务验证购买Token | 针对消耗型商品和非消耗型商品。在完成一笔支付后，解析出InAppPurchaseData中的purchaseToken，调用该接口获取购买数据及其签名数据，进一步验证购买信息的正确性。 |
| Subscription服务验证购买Token | 针对订阅型商品。验证购买Token，获取当前订阅状态。 |
| Order服务确认购买 | 服务端消耗商品，可替代客户端的consumeOwnedPurchase接口。 |
| 取消订阅 | 取消指定订阅。 |
| 延迟结算 | 延迟订阅结算。 |
| 返还订阅费用 | 返还指定订阅的费用。 |
| 撤销订阅 | 撤销指定订阅。 |
| 已取消或者退费购买查询 | 查询已取消或退费的购买记录。 |
| 应用购买记录相关支付订单查询 | 查询应用购买记录相关的支付订单。 |
| 接收订单/订阅关键事件通知 | 接收华为IAP服务器推送的订单/订阅关键事件通知（v1只支持订阅，v2支持订单和订阅）。 |
| 接收延迟付款型支付关键事件通知 | 接收延迟付款型支付的关键事件通知。 |
| 对返回结果验签 | 对IAP接口返回结果进行签名验证。 |
| （必要）验证InAppPurchaseData | 验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。 |

## 五、关键事件通知处理

来源：https://developer.huawei.com/consumer/cn/doc/HMSCore-Guides/notifications-about-subscription-events-0000001050035037

如果您的应用接入了华为IAP订单/订阅功能，建议您在开发者联盟配置订单/订阅通知地址，用于接收华为应用内支付服务器发送的关键事件通知。华为应用内支付关键事件通知支持v1（只支持订阅）和v2（支持订单和订阅）两个版本。

> 说明：建议所有提供一次性商品和订阅型商品的App均配置该通知地址，以便接收关键事件通知，为顾客提供更好、更及时的服务。另外，通知地址必须基于HTTPS并且配置有商业域名机构颁发的证书。

### 订单关键事件通知处理建议

您在收到IAP服务器的订单关键事件通知时，建议您根据通知内容中的购买Token，请求IAP服务器校验购买Token，并根据验证结果决定您是否需要发放商品。

订单通知仅作为订单状态发生变化的一个事件通知，由于网络延迟等导致您服务器收到通知事件有延迟，订单通知中携带的订单信息可能与IAP服务器的最新的订单状态不一致，建议不要根据订单通知来决定是否发放商品。

您在收到支付成功(1)通知类型后，建议根据如下订单通知处理流程进行业务逻辑处理：

1. IAP服务器发送订单关键事件通知。
2. 应用服务器收到通知请求后，从通知中获取购买Token。
3. 调用华为IAP服务器提供的Order服务验证购买Token接口，查询购买数据及签名数据。
4. IAP服务器返回购买数据及签名数据。为避免资金损失，您在验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。
5. 验证结果成功，处理发货，并记录购买商品的Token。请根据Order服务验证购买Token接口响应中InAppPurchaseData的purchaseState字段决定是否发货。若purchaseState为0，则执行发货操作。
6. 调用华为IAP服务器提供的Order服务确认购买接口确认购买（即消耗）。
7. IAP服务器返回确认购买结果。
8. 应用服务器返回通知响应给IAP服务器。

### 订阅关键事件通知处理建议

您在收到IAP服务器的订阅关键事件通知时，建议您根据通知内容中的购买Token，请求IAP服务器校验购买Token获取当前购买Token关联订阅的最新状态，并根据订阅最新状态决定您是否需要提供商品服务。

订阅通知仅作为订阅状态发生变化的一个事件通知，由于网络延迟等导致您服务器收到通知事件有延迟，订阅通知中携带的订阅信息可能与IAP服务器的最新的订阅状态不一致，建议不要根据订阅通知来决定是否提供商品服务。

您在收到INITIAL_BUY(0)、RENEWAL(2)、INTERACTIVE_RENEWAL(3)、RENEWAL_RECURRING(7)通知类型后，建议根据如下订阅通知处理流程进行业务逻辑处理，提供商品服务：

1. IAP服务器发送订阅关键事件通知。
2. 应用服务器收到通知请求后，从通知中获取购买Token。
3. 调用华为IAP服务器提供的Subscription服务验证购买Token接口，查询购买数据及其签名数据。
4. IAP服务器返回购买数据及其签名数据。为避免资金损失，您在验签成功后，必须校验InAppPurchaseData中的productId、price、currency等信息的一致性。
5. 校验订阅状态提供商品服务。请根据Subscription服务验证购买Token接口响应中InAppPurchaseData的subIsvalid字段决定是否发货。若subIsvalid为true，则执行发货操作。
6. 应用服务器返回通知响应给IAP服务器。
